Summary
핵심 MVP 백엔드 API 6개를 추가했습니다. GET /api/v1/stocks/search, GET /api/v1/stocks/{stock_code}, GET /api/v1/watchlist, POST /api/v1/watchlist, DELETE /api/v1/watchlist/{watchlist_id}, PATCH /api/v1/watchlist/{watchlist_id}/alert 가 라우터에 연결되었고, 화면 중심 응답을 반환하도록 구성했습니다. route handler는 가볍게 유지하고 실제 조합 로직은 서비스로 분리했습니다. 

공통 응답 구조 success, message, data, error_code 를 모든 핵심 API에 맞춰 적용했고, 비즈니스 에러와 요청 검증 에러도 같은 형태로 내려가도록 예외 핸들러를 추가했습니다. 그래서 앱 쪽에서는 성공/실패 처리 방식이 더 단순해집니다. 

종목 상세 API는 요청하신 최소 구조를 모두 포함하도록 만들었습니다. stock, price, status, levels, support_state, scenario, reason_lines, watchlist 를 전용 스키마로 분리했고, 상태 배지/시나리오/해설 3줄을 docs 명칭에 맞춰 최대한 유지했습니다. 

관심종목 API는 문서 요구사항에 맞춰 목록/추가/삭제/알림 ON/OFF 변경이 동작하도록 구현했습니다. 중복 추가, 없는 종목, 없는 관심종목 같은 경우도 명시적인 에러코드로 처리합니다. 

종목 상세의 가격 데이터를 안정적으로 내려주기 위해 daily_bars 모델과 Alembic 마이그레이션을 추가했습니다. 기존 stocks 모델과 연결도 넣어서 최신 가격/등락률/고가/저가/거래량을 화면 응답으로 만들 수 있게 했습니다. 

로컬에서 바로 확인할 수 있도록 최소 seed 데이터도 넣었습니다. 삼성전자, SK하이닉스, 가격 레벨, 지지 상태, 일봉 데이터, 데모 관심종목을 넣는 함수와 실행 스크립트를 추가했습니다. 

핵심 API 최소 테스트도 추가했습니다. 종목 검색, 종목 상세 구조, 관심종목 추가/조회/알림 변경/삭제 흐름을 각각 검증하도록 작성했습니다. 

실행/테스트 방법을 backend_api/README.md에 정리했습니다. 마이그레이션 → seed → 서버 실행 순서와 기본 테스트 명령을 바로 볼 수 있습니다.

구현 내용 설명
왜 서비스/리포지토리로 나눴나?
route 파일은 “요청 받기/응답 보내기”만 맡고, 실제 데이터 조합은 StockViewService, WatchlistService가 담당하게 했습니다. 이렇게 하면 나중에 Flutter나 admin_web이 붙어도 재사용하기 쉽고, 테스트도 쉬워집니다. 

종목 상세 API는 화면 기준으로 조합했습니다.
DB 테이블을 그대로 노출하지 않고, 앱 화면이 바로 쓰기 쉬운 형태로 묶었습니다. 예를 들면 상태 배지(status), 시나리오(scenario), 해설 3줄(reason_lines), 관심종목 여부(watchlist)를 한 번에 내려줍니다. 

인증이 아직 없어서 최소 MVP 방식으로 처리했습니다.
문서상 관심종목 API는 인증 필수인데, 현재는 로그인 시스템이 아직 없으므로 X-User-Identifier 헤더를 임시 사용자 식별자로 사용했습니다. 이건 MVP 진행을 위한 현실적인 연결 방식입니다. 나중에는 실제 사용자 인증으로 바꾸면 됩니다. 

가격 데이터가 없어서 상세 API가 비는 문제를 막기 위해 최소 일봉 테이블을 추가했습니다.
종목 상세와 관심종목 목록은 현재가/등락률/고가/저가/거래량이 꼭 필요해서, daily_bars를 추가해 화면 응답을 만들 수 있게 했습니다.

지지선 상태에 필요한 추적 필드들을 support_states에 확장하고, 신호 이력 저장용 signal_events, 홈/테마 구성을 위한 themes, theme_stock_maps, content_posts 모델과 마이그레이션을 추가했습니다. 이로 인해 상태엔진이 “처음 터치 시점, 테스트 저점, 이탈 여부, 재활용/무효 시점”까지 저장할 수 있게 되었습니다. 

상태 변화가 일어났을 때만 signal_events를 생성하고, support_state_id + signal_type + status_to 기반 signal_key로 동일 상태/동일 이벤트 중복 생성을 막는 로직을 넣었습니다. 그리고 이 엔진과 이벤트 생성을 한 번에 묶는 SupportMonitorService도 추가했습니다. 

GET /api/v1/home, GET /api/v1/stocks/{stock_code}/signals, GET /api/v1/themes API를 추가했고, 홈 API가 바로 Flutter 화면에 붙을 수 있도록 market_summary, featured_stocks, watchlist_signal_summary, themes, recent_contents를 한 번에 내려주도록 구성했습니다. 기존 종목 상세 API도 차트/연관 테마/연관 콘텐츠를 함께 주도록 보강했습니다.

상태엔진 설계상 가정한 부분

**이탈 기준은 기본 CLOSE_BREAK**로 두고, 설정 객체에서 LOW_BREAK로 바꿀 수 있게 했습니다. 다만 현재 PriceLevel에는 근접/반등 퍼센트만 저장되어 있어, 이탈 기준·재도달 기준·무효화 기준은 엔진 설정 레벨에서 관리합니다. 

무효화 기준은 MVP 단순화를 위해 “주요 고점 돌파 실패 + 테스트 기준 저점 종가 이탈”로 잡았습니다. 문서의 REBOUND_LOW_BREAK 권장 방향에 맞춘 최소 구현입니다.

Summary
백엔드에 관리자 운영 API를 추가해서 종목/가격 레벨 CRUD 성격 저장, 지지선 상태 조회 및 강제 수정, 신호 이벤트 조회, 홈 노출 관리, 테마 관리, 운영 로그 조회, 수동 푸시 저장까지 한 번에 처리할 수 있게 만들었습니다. 특히 상태 강제 수정과 수동 푸시는 모두 audit log로 남도록 연결했습니다.

알림/푸시 기반 구조를 백엔드에 추가했습니다. notifications, alert_settings, device_tokens, admin_audit_logs, home_featured_stocks 테이블을 위한 마이그레이션을 만들고, signal_event 발생 시 관심종목 사용자 기준으로 notification 저장과 stub push dispatch가 일어나도록 연결했습니다.

앱 API도 보강했습니다. 알림함 조회/읽음 처리, 알림 설정 조회/수정, 디바이스 토큰 저장 API를 추가했고 라우터에 연결했습니다.

홈 노출은 수동 운영 테이블을 우선 사용하고, 값이 없으면 기존 계산 기반 featured 로직으로 fallback 하도록 바꿨습니다. seed 데이터에도 홈 노출/알림/토큰/설정 샘플을 넣어 로컬 확인이 쉬워졌습니다.

Flutter 앱에 알림함 화면과 알림 설정 화면을 추가하고, 상단 벨 아이콘/마이 페이지에서 바로 진입할 수 있게 연결했습니다. 알림함은 읽음 처리까지 되고, 알림 설정은 서버 API와 직접 연동됩니다.

별도 admin_web을 새로 만들어, 예쁘기보다 운영 가능성 중심의 정적 관리자 화면을 추가했습니다. 여기서 대시보드, 종목/레벨 등록, 상태 강제 수정, 수동 푸시, 홈 노출, 테마, 운영 로그를 바로 다룰 수 있습니다.

README와 TODO 문서를 업데이트해서 실행 방법, 환경변수 예시, 현재 MVP 범위, 문서-코드 불일치(frontend_app/ 대신 루트 Flutter 앱, 실제 FCM 미연동 등)를 숨기지 않고 명확히 정리했습니다.

로컬 실행 방법
백엔드:

cd backend_api

가상환경 생성/활성화

pip install -e .

cp .env.example .env

alembic upgrade head

python scripts/seed_minimum_data.py

uvicorn app.main:app --reload.

Flutter 앱:

저장소 루트에서 flutter pub get

flutter run --dart-define=API_BASE_URL=http://127.0.0.1:8000/api/v1 --dart-define=USER_IDENTIFIER=demo-user.

관리자 웹:

cd admin_web

python3 -m http.server 4173

브라우저에서 http://127.0.0.1:4173 접속 후 API Base URL 입력.
