# KNOWN_ISSUES

이 문서는 현재 저장소 기준으로 **이미 구현되었지만 실서비스 출시 전 반드시 인지해야 하는 제한사항 / 외부 의존성 / 운영 리스크**를 정리한다.

목표는 “나중에 고치자”가 아니라, 출시 판단 시 빠뜨리기 쉬운 위험을 빠르게 확인하는 것이다.

---

## 1. 현재 알려진 핵심 이슈

### 1-1. Firebase / FCM 실전 검증은 외부 설정 완료가 필요함

현재 저장소에는 디바이스 토큰 등록, 알림 저장, FCM 발송 시도 코드가 존재한다.

하지만 아래가 완료되지 않으면 실전 푸시 검증은 끝난 것으로 볼 수 없다.

- Firebase 프로젝트 생성
- 플랫폼 앱 등록
- 실제 `FIREBASE_*` / `FCM_*` 값 주입
- 실기기 권한 허용 및 수신 검증

영향:
- 코드상 준비 완료와 실제 운영 푸시 성공은 별개다.
- 운영 출시 전 실기기 검증이 꼭 필요하다.

우선순위:
- Blocker 가능성 있음
- 출시 직전 반드시 실측 확인

### 1-2. 현재 기준 판정은 “로컬 MVP 완료”이며 “운영 배포 완료”와 다름

현재 README 와 TODO 문서 기준 판정은 로컬 실행/검수 가능한 MVP 완료판이다.

즉, 아래 항목은 별도 운영 준비가 남아 있을 수 있다.

- 운영 DB
- 운영 시크릿 분리
- HTTPS / CORS 운영값
- 운영 배포 절차
- rollback 점검

영향:
- 코드가 있어도 운영 환경 미설정이면 실제 출시 판정은 보류될 수 있다.

우선순위:
- Major

### 1-3. 관리자 웹은 정적 HTML/JS 기반이라 UX 보다 운영 속도에 초점이 맞춰져 있음

현재 admin_web 는 빠른 로컬 검수/운영 확인에는 적합하지만, 화면이 JSON 중심이라 장기 운영성은 추가 개선 여지가 있다.

영향:
- 숙련되지 않은 운영자에게는 사용성이 낮을 수 있다.
- 저장/검수 실수가 날 가능성이 있다.

우선순위:
- Minor

### 1-4. signal batch / push dispatcher 는 실행 루틴 관리가 중요함

배치와 dispatcher 스크립트는 존재하지만, 실제 운영에서는 아래가 확실해야 한다.

- 언제 실행하는지
- 중복 실행이 없는지
- 실패 로그를 누가 보는지
- 재시도 정책을 어떻게 적용하는지

영향:
- 실행 루틴이 정해지지 않으면 이벤트 누락/중복이 발생할 수 있다.

우선순위:
- Major

## 4. Cloud Firestore 읽기 비용 검토

현재 저장소는 Cloud Firestore를 사용하지 않는다.

따라서 known issues 관점에서도 Firestore read 비용 증가는 없다.

향후 Firestore를 도입하지 않는 한, 현재 문서 작업은 Firestore 읽기 소모와 무관하다.
