# Part11. 관리자 UI 전면 개편 - 운영 종목 / 지지저항 / 상태 관리 중심
버전: v1.0
대상: Codex / admin_web + backend 연동 정리
목적: 현재 관리자 화면은 운영자가 쓰기 어려운 raw form 중심 구조이므로, 종목 검색 + 지지/저항 입력 + 상태 확인/강제수정 중심의 실사용 UI로 전면 개편한다.

---

## 0. 이번 파트 한 줄 정의

운영자가 `stock_id,display_order,is_active` 같은 raw 텍스트를 입력하는 구조를 버리고,
`종목 검색 -> 지지선/저항선 입력 -> 상태 확인 -> 홈 추천/테마/콘텐츠 연결`
흐름으로 관리자 UI를 재설계한다.

---

## 1. 현재 문제 요약

### 1-1. 운영 UI가 개발자용 입력폼처럼 되어 있음
현재 문제:
- 홈 추천 저장: `stock_id,display_order,is_active`
- 테마 연결: `stock_id,role_type,score`
- 콘텐츠 연결: 관련 종목 ID / 관련 테마 ID
- 지지/저항 입력 UI 없음
- 지지선 상태 강제수정 UI 없음

이 구조는 운영자가 사용하기 어렵다.

### 1-2. 기능은 있는데 UI가 없다
백엔드에는 이미 아래 관리자 API가 존재한다.
- `GET /api/v1/admin/stocks`
- `POST/PUT /api/v1/admin/stocks`
- `GET /api/v1/admin/price-levels`
- `POST/PUT /api/v1/admin/price-levels`
- `GET /api/v1/admin/support-states`
- `PATCH /api/v1/admin/support-states/{state_id}/force`
- `GET/PUT /api/v1/admin/home-featured`
- `GET/POST/PUT /api/v1/admin/themes`
- `GET/POST/PUT /api/v1/admin/contents`

즉 백엔드 기능은 존재하지만, 운영 UI가 실사용 구조로 연결되지 않았다.

### 1-3. 운영 관심종목과 사용자 관심종목이 혼동된다
용어를 아래처럼 분리해야 한다.
- 운영 관심종목: Firebase adminWatchlist / 운영자가 관리하는 종목
- 내 관심종목: 앱 사용자의 개인 watchlist

---

## 2. 이번 파트 목표

1. 관리자 화면을 탭 기반 구조로 전면 개편
2. 종목 검색/자동완성 도입
3. 지지선/저항선 입력 UI 추가
4. 지지선 상태 조회/강제수정 UI 추가
5. 홈 추천/테마/콘텐츠 연결 시 raw ID 입력 제거
6. 운영자가 실제로 쓸 수 있는 수준으로 UX 단순화

---

## 3. 최종 IA(정보구조)

관리자 상단 탭을 아래 4개로 구성한다.

### 탭 A. 운영 종목
기능:
- 종목 검색 (종목코드/종목명)
- 종목 기본정보 조회/수정
- 홈 추천 포함 여부 확인
- 운영 메모 입력

### 탭 B. 지지/저항 관리
기능:
- 선택 종목의 지지선/저항선 목록 조회
- 지지선 추가/삭제
- 저항선 추가/삭제
- source_label(Firebase / operator) 표시

### 탭 C. 지지선 상태 관리
기능:
- support_states 목록 조회
- 상태 필터
- 종목 검색
- 상태 강제 수정
- 사유 입력

### 탭 D. 홈/테마/콘텐츠
기능:
- 홈 추천 종목 구성
- 테마 생성/수정
- 콘텐츠 생성/수정
- 종목/테마 연결은 검색 기반으로 처리

---

## 4. 탭별 상세 UI 요구사항

### 4-1. 운영 종목 탭
구성:
- 검색창: `종목코드 또는 종목명`
- 검색 결과 리스트
- 선택 종목 상세 카드
  - 종목코드
  - 종목명
  - 시장
  - 섹터
  - 테마 태그
  - 운영 메모
  - 앱 노출/활성
  - 홈 추천 포함 여부(읽기 또는 표시)

버튼:
- 저장
- 신규 종목 생성
- 초기화

### 4-2. 지지/저항 관리 탭
구성:
- 상단 검색창: `종목코드 또는 종목명`
- 선택 종목 요약 카드
- 지지선 리스트 카드
- 저항선 리스트 카드

지지선/저항선 항목 표시:
- 가격
- `Firebase` 또는 `수동` badge
- active 여부
- 삭제 버튼(수동 입력값만)
- Firebase imported 값은 기본적으로 읽기전용 badge 표시

하단 입력 박스:
- `지지선 값`
- `저항선 값`
- `추가`
- `저장`

행동 원칙:
- 한 종목에 지지선 여러 개 가능
- 한 종목에 저항선 여러 개 가능
- 저장 후 즉시 리스트 갱신

### 4-3. 지지선 상태 관리 탭
구성:
- 필터: 상태 드롭다운
- 검색창: 종목코드/종목명
- 상태 리스트 테이블

컬럼:
- 종목명
- 종목코드
- 지지선 가격
- 현재 상태
- 최근 업데이트
- 사유

행별 액션:
- `대기(WAITING)`
- `지지 확인중(TESTING_SUPPORT)`
- `직접 반등 성공(DIRECT_REBOUND_SUCCESS)`
- `돌파 후 재지지 성공(BREAK_REBOUND_SUCCESS)`
- `재사용(REUSABLE)`
- `무효(INVALID)`

상태 변경 시:
- 사유 입력 모달 표시
- `PATCH /api/v1/admin/support-states/{id}/force` 호출

### 4-4. 홈/테마/콘텐츠 탭
#### 홈 추천
현재 raw textarea 방식 제거.
대신:
- 종목 검색 후 추가
- 선택된 종목 카드 리스트
- 순서 조정(위/아래 버튼 또는 drag)
- 활성 on/off
- 저장

#### 테마
입력:
- 테마명
- 점수
- 요약
- 앱 노출/활성

연결 종목:
- 검색 후 추가
- 각 종목별 역할 선택 (`LEADER`, `FOLLOWER`)
- 점수 입력 가능

#### 콘텐츠
입력:
- 카테고리
- 제목
- 요약
- 외부 링크
- 썸네일 URL
- 앱 노출/공개
- 관련 종목 검색/선택
- 관련 테마 검색/선택

## 5. UX 원칙

### 필수 원칙
- 운영자에게 raw ID 입력 요구 금지
- 종목/테마 연결은 검색 기반
- Firebase imported 값은 시각적으로 구분
- 지지선 상태는 "자동 생성 + 필요 시 강제 수정" 구조로 안내
- 한 화면에 모든 걸 몰아넣지 말고 탭/섹션으로 분리

### 문구 원칙
- "운영 관심종목"과 "내 관심종목"을 명확히 구분
- "지지선 상태가 아직 준비되지 않았습니다"는 운영자 화면에서 원인/조치가 보이게 할 것
- 상태가 없는 종목은 "초기 상태 생성 필요" 또는 "배치 대기"로 표시

---

## 6. 백엔드 연동 원칙

### stocks
- 종목 검색 API 사용
- 기존 `GET /admin/stocks`

### price-levels
- 종목별 필터링 필요
- 현재 전체 목록 조회만 있다면 프론트 필터라도 우선 구현
- 가능하면 query param 기반 종목 필터 추가 권장

### support-states
- 상태 필터/종목 필터가 필요
- 현재 전체 조회 후 프론트 필터 가능
- 추후 서버 필터 추가 권장

### home-featured
- PUT 전체 교체 방식 유지 가능
- 프론트에서는 검색 선택 기반으로 payload 재구성

## 7. 강력 권장 추가 개선

### 7-1. /home 503 전체 실패 제거
백엔드 개선 권장:
- featured 종목 일부 support state 누락 시 홈 전체 503 금지
- 해당 종목만 status null 또는 `CALCULATING` 처리
- 테마/콘텐츠/나머지 종목은 정상 반환

### 7-2. 운영 관심종목 목록 API 명시화
가능하면 별도 목록 개념 추가:
- Firebase adminWatchlist 기반 운영 관심종목 API
- 앱/관리자에서 혼동 없이 사용 가능

### 7-3. support state 초기 상태 생성 버튼
관리자에서 아래 액션도 있으면 좋다.
- `초기 상태 다시 생성`
- `시그널 배치 재실행`
