## 1. 문서 목적

이 문서는 `srlabapp`의 **출시 가능 여부를 판단하는 기준 문서**다.

이 문서에서 정리하는 내용은 아래와 같다.

1. blocker / major / minor 분류 기준
2. 현재 저장소 기준 출시 판단
3. 1차 출시 범위(in scope)
4. 1차 출시 제외 범위(out of scope)
5. 출시 가능 최소 기준(release gate)
6. 실제 현재 상태 기준 known issue 요약

---

## 2. 현재 저장소 상태 한 줄 요약

현재 프로젝트는 **로컬 MVP 기능 흐름은 전반적으로 갖춰졌지만, 실운영 출시 관점에서는 외부 설정과 실기기 검증이 남아 있어 바로 프로덕션 출시 확정이라고 보기는 어렵다.**

즉, 판정은 아래처럼 나누는 것이 정확하다.

- 로컬/스테이징 기준: **출시 준비 진행 가능**
- 프로덕션 실출시 기준: **조건부 가능 또는 일부 blocker 해소 후 가능**

---

## 3. 이슈 분류 기준

### 3-1. Blocker

아래 중 하나라도 해당하면 출시를 보류한다.

- 앱 실행 직후 크래시 또는 핵심 탭 진입 불가
- 홈 / 검색 / 상세 / 관심종목 중 하나라도 완전히 동작하지 않음
- 관리자 로그인 또는 핵심 운영 입력이 불가능함
- 환경 분리가 불명확해서 dev/staging/prod 혼선이 발생할 수 있음
- migration 누락 또는 DB 불일치로 서비스 데이터 정합성이 깨짐
- signal batch 또는 push dispatcher 가 반복 실행 불가 수준으로 실패함
- 출시 범위에 포함된 푸시를 실제 운영에서 검증할 수 없음
- 민감정보 노출 또는 관리자 인증 우회 같은 보안상 치명점이 있음

### 3-2. Major

출시는 가능할 수 있지만, 빠른 수정 계획과 운영 통제가 필요하다.

- 핵심 기능은 되지만 특정 조건에서 오동작함
- 수동 운영으로 우회 가능하지만 자동화 흐름이 불안정함
- 실기기 푸시 검증 범위가 제한적임
- 차트/알림/상세 일부 데이터가 누락되면 UX 저하가 큼
- 관리자 role 분리 미구현처럼 운영 리스크가 남아 있음
- 장애 모니터링/로그 체계가 최소 수준에 머물러 있음

### 3-3. Minor

출시에 직접적인 차단은 아니지만 품질 개선이 필요하다.

- 문구/레이아웃/빈 상태 디자인 다듬기
- 카드 정렬, spacing, 시각 강조 부족
- 일부 화면의 안내 문구가 임시 수준
- 추가 필터/정렬/편의 기능 부재

## 4. 출시 가능 최소 기준

아래 항목을 모두 만족해야 1차 출시 가능으로 본다.

### 4-1. 사용자 핵심 흐름

- [ ] 홈 화면 정상 조회
- [ ] 종목 검색 가능
- [ ] 종목 상세 진입 가능
- [ ] 차트 최소 렌더링 가능
- [ ] 관심종목 등록/해제 가능
- [ ] 알림함 조회 및 읽음 처리 가능

### 4-2. 운영 핵심 흐름

- [ ] 관리자 로그인 가능
- [ ] 종목/레벨/테마/콘텐츠 수정 가능
- [ ] 관리자 입력 후 앱/공개 API 반영 가능
- [ ] 상태 강제 수정과 감사 로그 확인 가능

### 4-3. 데이터/배치 흐름

- [ ] health endpoint 정상
- [ ] signal batch dry-run 성공
- [ ] signal batch 실제 실행 성공
- [ ] signal_event 생성 확인
- [ ] notification 생성 확인
- [ ] dispatcher 실행 및 상태 갱신 확인

### 4-4. 배포/환경 분리

- [ ] dev / staging / prod 환경 구분 가능
- [ ] 운영용 비밀정보는 배포 환경에서만 주입하고, 저장소에는 `backend_api/.env.example` 만 남긴다
- [ ] 릴리즈 빌드에 dev URL 미포함 검증 완료
- [ ] 실제 운영 DB/FCM 자격증명 주입 계획이 문서화됨

### 4-5. 푸시 관련 최소 기준

아래 둘 중 하나는 반드시 충족해야 한다.

1. **푸시를 1차 출시 범위에 포함한다면**: Android 실기기 기준 최소 1회 이상 실수신 검증 완료  
2. **푸시를 운영 준비 기능으로 남긴다면**: 알림 DB 저장 + 알림함 노출 + fallback 로그까지는 검증 완료

## 5. 현재 프로젝트 기준 분류 결과

### 5-1. Blocker 후보

#### B-1. 실제 Firebase/FCM 실운영 검증 미완료
- 현재 코드는 FCM 연동 가능 상태지만, 실제 Firebase 콘솔/키/실기기 수신 검증은 외부 단계로 남아 있다.
- 푸시를 출시 핵심 기능으로 약속한다면 이 항목은 blocker 다.

#### B-2. 운영 배포 자격증명 및 인프라 분리 미완료
- 운영용 DB, 관리자 비밀키, HTTPS/CORS, 스토어 릴리즈 서명 등은 TODO 로 남아 있다.
- 실제 프로덕션 출시에 바로 들어간다면 blocker 다.

### 5-2. Major

#### M-1. 관리자 권한이 단일 관리자 계정 중심
- 현재는 환경변수 기반 관리자 인증은 있으나 세분화된 role 관리가 없다.
- 운영자 수가 늘어나면 책임 추적/권한 제한이 약하다.

#### M-2. push / batch 운영 자동화 고도화 부족
- 수동 실행은 가능하지만 운영 스케줄러, 장애 알림, 재시도 체계는 고도화 여지가 크다.

#### M-3. 앱 일부 화면은 MVP 안내 문구 수준
- 마이 화면의 공지/문의 안내는 운영 준비 전용 안내 문구로 정리되었지만, 실제 연결 기능은 아직 없다.
- 핵심 거래 흐름은 아니지만 출시 시 사용자 기대 관리가 필요하다.

### 5-3. Minor

#### N-1. 디자인/문구/빈 상태 개선
- 홈/알림/테마/마이 화면은 기능 중심 MVP 품질이며, 완성형 UX 고도화는 후속 작업이다.

#### N-2. 세부 필터/정렬/운영 대시보드 고도화
- 지금도 운영 가능하지만 대량 운영 편의성은 더 다듬을 수 있다.

## 6. 현재 출시 판정

### 6-1. 로컬 MVP / staging 판정

**판정: 출시 준비 가능**

근거:
- 핵심 API, 관리자 API, 배치, notification 저장, Flutter 기본 흐름, 테스트가 이미 존재한다.
- 문서와 실제 저장소 구조도 대체로 정렬되어 있다.

### 6-2. 실제 프로덕션 판정

**판정: 조건부 가능, 아래 blocker 해소 전에는 최종 출시 보류 권장**

출시 전 필수 확인:
- Firebase 실기기 수신 검증
- 운영 자격증명/환경 분리 완료
- Android/iOS 릴리즈 빌드 검증
- `python backend_api/scripts/check_release_readiness.py` 실행 완료

## 7. 1차 출시 범위(In Scope)

이번 1차 출시에서 반드시 책임지고 제공할 범위는 아래다.

### 7-1. 앱
- 홈
- 관심종목
- 종목 검색
- 종목 상세
- 테마 목록/상세
- 알림함
- 알림 설정
- 최소 쇼츠/콘텐츠 노출

### 7-2. backend
- health endpoint
- 홈/검색/상세/테마/콘텐츠/관심종목/알림 API
- support state engine
- signal event 생성
- notification 저장
- device token 등록

### 7-3. admin
- 관리자 로그인
- 종목 CRUD 성격 입력
- 가격 레벨 입력
- 상태 조회/강제 수정
- 신호 이벤트 조회
- 홈 노출 관리
- 테마 관리
- 콘텐츠 관리
- 수동 푸시 저장
- 감사 로그 조회

### 7-4. 운영
- signal monitor 수동 실행 절차
- notification dispatcher 수동 실행 절차
- 환경 분리 가이드
- known issues 문서 기반 위험 통제

---

## 8. 1차 출시 제외 범위(Out of Scope)

아래는 문서와 현재 구현 상태를 기준으로 1차 출시에서 제외하거나 후순위로 본다.

- 초단위 실시간 전종목 스트리밍
- 고급 차트 인터랙션
- 복잡한 사용자 권한 체계
- 결제 / 구독 / 과금
- 커뮤니티 / 댓글 / 소셜 기능
- 고급 추천 엔진
- 대규모 운영 통계 대시보드
- 완전 자동화된 배치 장애 복구 체계
- 다국어 대응

## 9. 출시 승인 권장 규칙

### 9-1. 출시 가능

아래를 모두 만족하면 출시 승인 가능으로 본다.

- blocker 0건
- major 이슈별 우회/운영 통제 방법 문서화
- QA 체크리스트 완료
- known issues 최신화
- 운영자 1인 이상이 daily / weekly 운영 절차를 이해함

### 9-2. 조건부 출시 가능

아래와 같으면 제한적 출시만 검토한다.

- 핵심 앱 흐름은 정상
- push 실기기 검증 또는 운영 인프라 일부가 미완료
- 사용자 수를 제한한 soft launch / 내부 검수 배포로 먼저 확인 가능

### 9-3. 출시 보류

아래 중 하나면 출시 보류다.

- blocker 존재
- 운영 환경 혼선 가능성 존재
- 관리자 수정이 앱에 반영되지 않음
- signal / notification 흐름이 끊김

---

## 10. 권장 최종 판정

현 시점의 가장 현실적인 권장안은 아래다.

1. **내부/스테이징 QA는 바로 진행**한다.
2. **soft launch 기준으로는 조건부 진입 가능**하다.
3. **대외 프로덕션 정식 출시 전에는 FCM 실기기 검증과 운영 비밀정보 분리를 끝낸다.**

---

## 11. Cloud Firestore 읽기 비용 검토

현재 프로젝트는 Cloud Firestore를 사용하지 않는다.

따라서 출시 준비 문서를 보강해도 **Firestore read 비용 영향은 없다**.
운영 비용은 PostgreSQL 조회, batch 실행 빈도, notification dispatch 재시도, FCM 전송량 기준으로 관리하면 된다.
