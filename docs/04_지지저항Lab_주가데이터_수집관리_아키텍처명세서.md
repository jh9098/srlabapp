# 1. 문서 목적

본 문서는 지지저항Lab 앱의 주가데이터 처리 구조를 정의하기 위한 문서이다.

지지저항Lab 앱은 일반적인 HTS/MTS처럼 모든 실시간 체결 정보를 그대로 보여주는 앱이 아니다.  
본 앱의 핵심은 **지지선/저항선 기반 상태 판단**, **관심종목 관찰**, **가격 근접 신호**, **짧은 해설 제공**이다.

따라서 주가데이터 시스템은 아래 목적을 만족해야 한다.

1. 가격 데이터를 안정적으로 수집할 수 있어야 한다.
2. 앱에 필요한 형태로 데이터를 가공할 수 있어야 한다.
3. 지지선/저항선 상태판정 엔진이 사용할 수 있어야 한다.
4. 관심종목 중심의 실시간성은 확보하되, 과도한 인프라 복잡도는 피해야 한다.
5. 운영자가 직접 레벨과 신호를 관리할 수 있는 구조와 연결되어야 한다.

---

# 2. 핵심 원칙

## 2-1. 데이터 수집과 앱 노출은 분리한다
주가데이터는 백엔드가 수집하고, 앱에는 가공된 결과를 제공한다.  
앱이 직접 외부 시세 API를 호출하는 구조는 사용하지 않는다.

---

## 2-2. 가격 원문보다 신호 중심으로 제공한다
지지저항Lab 앱의 핵심은 초단위 시세 원문이 아니라,  
현재 가격이 지지선/저항선과 어떤 관계에 있는지 판단한 결과다.

즉, 앱에는 아래와 같은 데이터가 더 중요하다.

- 현재가
- 지지선까지 거리
- 저항선까지 거리
- 지지선 접근 여부
- 지지선 이탈 여부
- 반등 성공 여부
- 재활용 가능 여부
- 짧은 해설 문구

---

## 2-3. 초기에는 관심종목 중심으로 운영한다
MVP 단계에서는 전종목 실시간 추적보다 아래 종목 위주로 운영한다.

1. 사용자의 관심종목
2. 운영자가 지정한 오늘의 관찰 종목
3. 주요 테마 대표 종목

---

## 2-4. 실시간성과 안정성의 균형을 잡는다
지지저항Lab은 초단타 매매앱이 아니므로 초저지연 시스템이 필수는 아니다.  
중요한 것은 **이벤트 발생 시 빠르게 상태를 갱신하고 푸시를 발송하는 것**이다.

---

## 2-5. 운영 통제를 우선한다
초기에는 완전 자동화보다 운영자가 직접 개입 가능한 구조를 우선한다.  
지지선/저항선 값은 관리자에서 입력/수정 가능해야 하며, 신호 상태도 운영자가 검토할 수 있어야 한다.

---

# 3. 시스템 목표

본 주가데이터 시스템의 목표는 아래와 같다.

## 3-1. 데이터 수집
- 종목별 시세를 안정적으로 수집한다.
- 장중 현재가 및 필요 시 실시간 가격 흐름을 추적한다.
- 장마감 후 일봉 데이터를 정리한다.

## 3-2. 데이터 저장
- 앱 차트 표시용 데이터를 저장한다.
- 상태판정 엔진에 필요한 가격 이력을 저장한다.
- 실시간 가격은 캐시에, 과거 봉 데이터는 DB에 저장한다.

## 3-3. 데이터 가공
- 지지선/저항선과 현재가의 거리 계산
- 지지선 상태판정
- 관심종목 상태 요약
- 홈 화면용 카드 데이터 생성
- 테마/브리핑용 요약 데이터 생성

## 3-4. 데이터 제공
- 앱 API로 필요한 가공 결과를 제공한다.
- 관리자 화면에서도 같은 데이터를 활용할 수 있어야 한다.

# 4. 전체 아키텍처 개요

전체 구조는 아래 흐름을 따른다.

```text
외부 시세 데이터 소스
    ↓
수집 모듈
    ↓
정규화 모듈
    ↓
실시간 캐시 / 영구 저장소
    ↓
신호 계산 엔진
    ↓
콘텐츠 가공 레이어
    ↓
앱 API / 관리자 API / 푸시 시스템
5. 주요 구성 요소
5-1. 외부 시세 데이터 소스

외부 시세 데이터 소스는 다음 역할을 한다.

종목 기본 정보 제공

현재가 제공

장중 가격 흐름 제공

일봉/분봉 데이터 제공

초기 MVP에서는 공식 시세 API 또는 합법적으로 사용 가능한 데이터 소스를 사용한다.
구체적 공급자는 운영 환경에 따라 교체 가능하도록 추상화한다.

5-2. 수집 모듈

수집 모듈은 외부 데이터 소스로부터 데이터를 받아오는 역할을 한다.

주요 책임:

현재가 조회

장중 가격 수신

일봉 수집

분봉 수집 또는 분봉 생성용 원천 수집

실패 시 재시도

호출 제한 관리

5-3. 정규화 모듈

외부 데이터 형식은 공급자마다 다를 수 있으므로 내부 표준 형식으로 변환해야 한다.

주요 책임:

종목코드 표준화

날짜/시간 형식 표준화

가격 필드명 통일

거래량 필드명 통일

누락값 처리

이상값 필터링

5-4. 실시간 캐시

장중 자주 바뀌는 데이터는 캐시에 저장한다.

캐시 저장 대상 예시:

현재가

장중 고가/저가

누적 거래량

최근 업데이트 시각

지지선/저항선 거리

현재 상태 배지용 데이터

초기 권장:

Redis 또는 동등한 인메모리 캐시 사용

5-5. 영구 저장소

과거 데이터와 이력 데이터는 영구 저장소에 저장한다.

저장 대상 예시:

종목 마스터

일봉 데이터

분봉 데이터

지지선/저항선

지지선 상태 이력

신호 이벤트

알림 이력

테마 연결 정보

초기 권장:

PostgreSQL 또는 동등한 관계형 DB 사용

5-6. 신호 계산 엔진

수집된 가격 데이터와 운영자가 지정한 레벨 데이터를 바탕으로 상태를 계산한다.

주요 책임:

지지선 근처 진입 판정

지지선 이탈 판정

직접 반등 성공 판정

이탈 후 반등 성공 판정

재활용 가능 판정

무효 판정

저항선 관련 상태 판정(추후 확장)

상태 변경 이벤트 기록

5-7. 콘텐츠 가공 레이어

앱은 수치 원문보다 가공 결과가 중요하므로, 사용자에게 보여줄 데이터를 생성한다.

주요 책임:

홈 카드 데이터 생성

관심종목 상태 요약 생성

종목 상세 해설 데이터 생성

테마 카드 데이터 생성

푸시 문구 생성

5-8. API 레이어

앱과 관리자 화면은 직접 DB를 읽지 않고 API를 통해 데이터를 조회한다.

주요 책임:

관심종목 API 제공

신호 이벤트 API 제공

관리자 API 제공

상태 변화가 발생했을 때 사용자에게 푸시 알림을 발송한다.

11. 데이터 정합성 원칙
11-1. 시간 기준 일관성

모든 시간 데이터는 내부 표준 시간대를 기준으로 저장한다.

권장:

DB 저장은 UTC 또는 명확한 시간대 포함 저장

앱 응답 시 사용자 지역 시간으로 변환 가능

11-2. 종목코드 일관성

외부 소스 종목코드 형식이 달라도 내부 표준코드로 통일한다.

11-3. 동일 시점 중복 저장 방지

같은 종목, 같은 시점, 같은 봉 데이터는 중복 저장하지 않는다.

11-4. 데이터 누락 시 보수적 처리

가격 데이터가 누락되면 상태를 함부로 변경하지 않는다.
필요 시 판정 유예 또는 데이터 부족 상태를 둔다.

14. 캐시 전략
14-1. 현재가 캐시

장중 자주 변하는 데이터는 짧은 TTL 또는 실시간 갱신으로 유지한다.

14-2. 홈 데이터 캐시

홈은 매 요청마다 무거운 계산을 하지 않도록 캐시를 둘 수 있다.

예:

10초~60초 단위 갱신

장마감 이후 더 긴 캐시 허용

14-3. 종목 상세 캐시

차트 데이터는 자주 변하지 않으므로 일부 캐싱 가능하다.
다만 현재가/상태는 별도 최신 데이터와 합쳐서 응답한다.

14-4. 테마 데이터 캐시

테마는 초단위 변동보다 분단위/장마감 후 갱신이 더 적합하다.

16. 운영자 개입 포인트

초기 MVP에서는 아래 항목에 운영자 개입을 허용한다.

16-1. 종목 활성/비활성 관리

관심 추적 대상 종목을 운영자가 제어할 수 있다.

16-2. 지지선/저항선 직접 입력

자동 추출보다 수동 등록을 우선한다.

16-3. 신호 상태 검토

REUSABLE, INVALID 등 핵심 상태는 운영자가 확인 가능해야 한다.

16-4. 이벤트 무시 처리

잘못 생성된 이벤트는 운영자가 무시/보정할 수 있어야 한다.

17. 확장 가능한 구조 원칙
17-1. 데이터 소스 교체 가능

외부 API 제공자가 바뀌어도 수집 모듈만 교체하면 되도록 설계한다.

17-2. 전종목 확장 가능

초기에는 관심종목 중심이지만, 장기적으로 전종목 확장 가능한 구조를 염두에 둔다.

17-3. 저항선 엔진 확장 가능

현재 문서는 지지선 중심이지만, 이후 저항선 상태관리 엔진과 쉽게 연결돼야 한다.

17-4. 테마 엔진 확장 가능

뉴스/테마/거래대금 기반 랭킹과 연결할 수 있도록 구조를 닫지 않는다.

22. 최종 요약

지지저항Lab의 주가데이터 시스템은
외부 시세를 받아 앱에 필요한 상태/신호 중심 데이터로 가공해 제공하는 구조를 기준으로 설계한다.

핵심은 아래와 같다.

앱이 직접 시세를 받지 않고 백엔드가 수집한다.

장중에는 관심종목 중심으로 현재가와 상태를 관리한다.

장마감 후에는 일봉 기준으로 재정리하고 상태를 재검토한다.

실시간 원문보다 지지선/저항선 기반 상태판정 결과가 중요하다.

초기에는 운영자 수동 통제를 허용하고, 이후 자동화를 확장한다.

즉, 본 아키텍처의 목적은
**“가격 데이터 그 자체를 보여주는 것”이 아니라, 가격 데이터를 기반으로 사용자가 이해하기 쉬운 신호와 상태를 제공하는 것”**이다.
