## 1. 문서 목적

이 문서는 출시 후 실제 운영 기준으로 무엇을 먼저 개선할지 우선순위를 정리한 문서다.

원칙은 아래와 같다.

1. 출시 직후에는 새 기능보다 안정화가 우선이다.
2. 운영자가 수동으로 버틸 수 없는 영역부터 자동화한다.
3. MVP 범위를 벗어나는 대형 기능은 3순위 이하로 미룬다.

---

## 2. 우선순위 기준

### 2-1. 1순위
- 출시 안정성에 직접 영향
- 사용자 불만과 이탈 가능성이 큼
- 운영자가 매일 수동으로 막고 있으면 안 되는 항목

### 2-2. 2순위
- 핵심 기능은 유지되지만 사용성/반복 사용성을 높이는 항목
- 운영 효율과 신뢰도를 같이 개선하는 항목

### 2-3. 3순위
- MVP 이후 확장 가치가 크지만 당장 없어도 서비스 운영은 가능한 항목

---

## 3. 1순위 로드맵

### 3-1. 안정화 / 운영 필수

1. **FCM 실운영 검증 완료**
   - Android/iOS 실기기 수신 검증
   - foreground/background/terminated 동작 검증
   - invalid token 처리 운영 규칙 확정

2. **운영 환경 비밀정보 분리**
   - 운영 DB 계정 분리
   - 관리자 계정/비밀번호 강한 값으로 교체
   - `ADMIN_JWT_SECRET`, `SECRET_KEY`, FCM 자격증명 분리

3. **릴리즈 빌드 검증**
   - Android 서명
   - iOS 배포 설정
   - dev URL 미포함 확인

4. **운영 smoke test 체계 고정**
   - 배포 직후 health check
   - 홈/상세/관심종목/알림함 점검
   - signal batch / dispatcher 수동 실행 점검

5. **치명 버그 핫픽스 대응 체계**
   - known issue 업데이트
   - rollback 절차 문서 확인
   - 배치/푸시 장애 시 비활성화 기준 정리

### 3-2. 데이터 신뢰도 개선

6. **signal rule 튜닝**
   - false positive / false negative 점검
   - REUSABLE / INVALID 판정 사례 수집
   - 운영자 메모와 실제 이벤트 비교

7. **운영 로그 / 실패 로그 가시성 강화**
   - dispatcher 실패 비율 확인
   - signal batch 에러 원인 모니터링
   - 관리자 주요 수정 이력 점검 루틴 추가

## 4. 2순위 로드맵

### 4-1. 앱 UX 개선

1. 홈 화면 요약 구조 개선
2. 알림함 가독성 개선
3. 테마/콘텐츠 탐색 UX 개선
4. 관심종목 정렬/필터 강화
5. 차트 시각 품질 향상

### 4-2. 운영 효율 개선

6. 관리자 권한(role) 분리
7. 관리자 입력 폼 검증 강화
8. 운영 대시보드 지표 추가
9. 콘텐츠/테마/홈 노출 편집 편의 향상
10. 수동 푸시 대상 선택 편의 개선

### 4-3. 품질 체계 보강

11. Flutter integration test 또는 화면 단위 테스트 확대
12. backend smoke script 추가
13. 배포 후 자동 점검 스크립트 보강

---

## 5. 3순위 로드맵

### 5-1. 제품 확장

1. 알림 세분화
2. 사용자 개인화
3. retention 기능 강화
4. 운영 통계/성과 대시보드
5. 프리미엄/구독 기능 검토

### 5-2. 고도화 기능

6. 더 정교한 테마 랭킹
7. 콘텐츠 추천 개선
8. 관리자 분석 화면 고도화
9. 시장 요약 자동 생성 품질 향상
10. 실시간성 고도화 여부 재검토

---

## 6. 권장 일정 예시

### 출시 후 1주
- 1순위 항목 중 푸시/배포/환경 분리 점검
- 치명 버그 핫픽스 준비
- daily 운영 체크 루틴 정착

### 출시 후 2~4주
- signal rule 튜닝
- 운영 로그/실패 로그 가시성 강화
- 알림함/홈 UX 개선 시작

### 출시 후 1~2개월
- 관리자 권한 분리
- 관심종목/테마 UX 개선
- 테스트 자동화 범위 확대

### 출시 후 2개월+
- 개인화 / retention / 통계 / 수익화 검토

## 7. 지금 바로 착수할 1순위 작업 예시

비전공자 기준으로도 바로 실행 가능한 실제 작업 예시는 아래다.

1. Firebase 실제 프로젝트를 만들고 Android 실기기 1대로 푸시 수신을 검증한다.
2. `backend_api/.env` 를 운영용 값과 개발용 값으로 분리한다.
3. staging 환경에서 `signal monitor` 와 `notification dispatcher` 를 하루 1회 이상 수동 검수한다.
4. 관리자 비밀번호를 기본값에서 강한 값으로 바꾼다.
5. known issues 문서를 보면서 출시 당일 대응 우선순위를 정한다.

---

## 8. 로드맵 운영 원칙

- 1순위가 끝나기 전에는 대형 신규 기능을 넣지 않는다.
- 운영자가 매일 수동 확인해야 하는 항목은 먼저 자동화하거나 점검 도구를 붙인다.
- 사용자가 체감하는 신뢰도는 신호 정확도와 푸시 품질에서 가장 크게 결정된다고 본다.

---

## 9. Cloud Firestore 읽기 비용 검토

현재 프로젝트는 Cloud Firestore를 사용하지 않는다.

따라서 이 로드맵에서 제안한 작업들도 **Firestore read 비용 증가와 무관**하다.
향후에도 Firestore를 도입하지 않는 한, 현재 운영 비용 검토 대상은 PostgreSQL/FCM/배치 실행량이다.
