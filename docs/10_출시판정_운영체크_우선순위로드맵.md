버전: v0.1  
작성일: 2026-03-20  
상태: 초안  
목적: 현재 저장소의 실제 구현 상태를 기준으로 1차 출시 범위, 제외 범위, 운영 체크리스트, blocker/major/minor 분류 기준, 출시 후 우선순위 로드맵을 정리한다.

## 1. 현재 프로젝트 상태 요약

현재 `srlabapp` 저장소는 **로컬 실행/검수 가능한 MVP 완료판**을 목표로 정리된 상태다.

현재 확인 가능한 구현 범위는 아래와 같다.

- Flutter 앱 핵심 화면 존재
- FastAPI 앱/관리자 API 존재
- admin_web 로그인 및 운영 화면 존재
- signal batch 실행 스크립트 존재
- notification dispatcher 실행 스크립트 존재
- 핵심 backend 테스트 세트 존재
- 운영 배포/환경분리 문서 존재
- TODO 문서는 주로 프로덕션 외부 설정 잔여 항목 중심으로 정리되어 있음

즉, 현재 상태는 **로컬 MVP 기능 검수 가능 단계**로 보는 것이 맞고, 실서비스 출시 여부는 아래 출시 가능 최소 기준과 외부 설정 완료 여부를 함께 봐야 한다.

## 2. 1차 출시 범위 (In Scope)

### 2-1. 사용자 앱 기능

1차 출시 범위에 포함하는 기능은 아래와 같다.

1. 홈 화면
   - 오늘의 관찰 종목
   - 관심종목 신호 요약
   - 테마 / 콘텐츠 요약 노출
2. 관심종목 기능
   - 등록
   - 삭제
   - 알림 on/off
3. 종목 검색
4. 종목 상세
   - 현재 상태 배지
   - 가격 정보
   - 차트
   - 지지선 / 저항선 요약
   - 최근 신호 카드
   - 시나리오 / 해설 3줄
   - 관련 테마 / 관련 콘텐츠
5. 테마 화면
6. 쇼츠 최소 화면
7. 마이 화면
8. 알림함 / 알림 설정
9. 디바이스 토큰 등록 및 푸시 라우팅 기반

### 2-2. 운영/백엔드 기능

1. 홈 / 종목 / 관심종목 / 알림 / 테마 API
2. 관리자 로그인 및 Bearer 토큰 인증
3. 종목 / 가격 레벨 / 상태 / 이벤트 / 홈 노출 / 테마 / 콘텐츠 운영 API
4. signal batch 기반 신호 자동 생성
5. notifications 저장
6. push dispatcher 기반 pending 알림 처리
7. 감사 로그 조회

## 3. 1차 출시 제외 범위 (Out of Scope)

아래 항목은 현재 MVP 문서 기준으로 1차 출시 범위에서 제외한다.

9. 고급 통계 대시보드
10. 정교한 콘텐츠 CMS 확장 기능

## 4. blocker / major / minor 분류 기준

### 4-1. Blocker 기준

출시를 멈춰야 하는 기준이다.

- 앱 진입 불가 또는 크래시 반복
- 홈 / 검색 / 상세 / 관심종목 중 하나라도 완전 불능
- 관리자 로그인 불가
- backend 핵심 API 장애 지속
- signal batch 또는 push dispatcher 실행 불가
- 운영 환경 분리 실패
- 민감정보 노출 또는 심각한 보안 결함

### 4-2. Major 기준

출시는 가능하나 사용자 핵심 경험이나 운영 안정성을 크게 깎는 기준이다.

- 핵심 기능이 간헐적으로 실패함
- 특정 화면에서 데이터 누락 시 UX 가 크게 깨짐
- 관리자 저장 후 앱 반영이 불안정함
- 푸시 일부 누락 / 재시도 정책 미흡
- 배치 로그와 운영 추적성이 부족해 원인 파악이 어려움

### 4-3. Minor 기준

출시 차단은 아니지만 빠르게 품질을 높여야 하는 기준이다.

- 문구 개선
- 화면 정렬/간격 개선
- 빈 상태 / 에러 상태 문구 보강
- 관리자 UI 사용성 개선
- 로그 메시지 가독성 개선

## 5. 출시 가능 최소 기준

현재 프로젝트의 1차 출시 가능 최소 기준은 아래와 같다.

### 5-1. 필수 통과 항목

- 홈 화면 정상 로드
- 종목 검색 정상 동작
- 종목 상세 및 차트 정상 노출
- 관심종목 등록/해제/알림 설정 정상 동작
- 알림함 조회 및 읽음 처리 정상 동작
- 관리자 로그인 및 주요 운영 저장 정상 동작
- signal batch 수동 실행 성공
- push dispatcher 수동 실행 성공
- 관리자 수정 → 앱 반영 확인
- 치명적 크래시 없음
- 환경 분리 설정 확인 완료

### 5-2. 실전 출시 전 추가 조건

아래는 저장소 코드만으로는 자동 충족되지 않으므로 실제 운영 준비에서 반드시 닫아야 한다.

- 운영 DB 준비
- 운영 admin 계정/비밀번호/시크릿 분리
- FCM 실제 자격증명 연결
- 실제 기기 푸시 수신 검증
- release 빌드가 prod API 를 바라보는지 최종 재확인
- 배포 후 smoke test / rollback 절차 점검

## 6. 일일 운영 체크리스트

매 영업일 기준으로 아래 순서대로 점검한다.

### 6-1. 오픈 전

- [ ] `/health` 와 `/api/v1/health` 응답 확인
- [ ] 전일 signal batch / notification dispatcher 실패 로그 확인
- [ ] 운영 대상 종목/레벨 비정상 수정 여부 확인
- [ ] 홈 추천 종목 / 테마 / 콘텐츠 노출 데이터 확인
- [ ] 관리자 로그인 정상 여부 확인
- [ ] 푸시 설정/키 변경 여부 확인

### 6-2. 장중

- [ ] 주요 관찰 종목 상태 변화 이상치 모니터링
- [ ] signal_event 생성량 급증/급감 여부 확인
- [ ] pending notification 누적 여부 확인
- [ ] invalid token / failed notification 비율 확인
- [ ] 운영자가 수동 수정한 상태값 메모 확인

### 6-3. 마감 후

- [ ] signal batch 실행 결과 확인
- [ ] 중복 이벤트 비정상 증가 여부 확인
- [ ] INVALID / REUSABLE 전환 종목 검토
- [ ] 홈 노출 / 테마 / 콘텐츠 다음날 반영 필요 항목 정리
- [ ] 주요 이슈를 blocker / major / minor 로 분류해 기록

## 8. 출시 후 우선순위 로드맵

### 8-1. 1순위 로드맵 (출시 직후 1~2주)

가장 현실적으로 빨리 닫아야 하는 항목들이다.

1. 실기기 푸시 수신 안정화
2. 운영 로그/실패 로그 점검 루틴 고정
3. 빈 상태 / 네트워크 오류 상태 UX 보강
4. 관리자 저장 후 반영 검수 루틴 정례화
5. blocker / major 이슈 빠른 핫픽스 체계 정리

### 8-2. 2순위 로드맵 (출시 후 2~4주)

사용성/운영성 개선 중심이다.

1. 관리자 UI 가독성 개선(JSON 중심 화면 보완)
2. 알림 문구/상세 설명 품질 개선
3. 테마/콘텐츠 관리 흐름 보강
4. 배치/푸시 처리 결과 대시보드 보강
5. 운영자 수동 수정 메모/이력 추적성 강화

### 8-3. 3순위 로드맵 (출시 후 1~2개월)

MVP 범위는 유지하면서 다음 단계 기반을 다지는 항목이다.

1. staging/prod 운영 루틴 자동화 보강
2. 역할(Role) 분리 확장 검토
3. 더 촘촘한 장애 알림/모니터링 체계 도입
4. 차트/상세 화면 정보 구조 개선
5. 테마/쇼츠/콘텐츠 연계 강화

## 9. 문서 운영 원칙

- blocker / major / minor 분류는 감이 아니라 실제 재현 가능성과 사용자 영향도로 판단한다.
- 출시 가능 여부는 “기능이 많으냐”가 아니라 “핵심 플로우가 안정적으로 도는가”로 판단한다.
- 운영 문서는 한 번 쓰고 끝내지 말고, 실제 장애/수정 이력을 반영해서 매주 갱신한다.

---

## 10. Cloud Firestore 읽기 비용 검토

현재 저장소는 Cloud Firestore를 사용하지 않는다.

따라서 이번 출시판정/운영체크/로드맵 문서 정리로 인해 Firestore read 비용 증가는 없다.

현재 읽기/운영 비용 관점에서 더 중요한 것은 PostgreSQL 조회, backend API 호출량, 푸시 재시도량, 배치 중복 실행 방지다.
