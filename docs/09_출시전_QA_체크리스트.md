## 1. 문서 목적

이 문서는 출시 직전 최종 QA 담당자가 아래 질문에 바로 답할 수 있도록 만든 실행 문서다.

- 지금 앱을 1차 출시 가능한가?
- 어떤 항목을 반드시 실제 기기/운영 환경에서 다시 확인해야 하는가?
- blocker / major / minor 이슈를 어떻게 분류할 것인가?
- 어느 항목을 통과해야 최소 출시 기준을 만족하는가?

본 문서는 기획 설명 문서가 아니라 **실행용 체크리스트**다.

## 2. 현재 저장소 기준 QA 대상 범위

현재 저장소 기준으로 QA 범위에 포함되는 대상은 아래와 같다.

1. Flutter 앱
   - 홈
   - 관심종목
   - 종목 검색
   - 종목 상세
   - 테마
   - 쇼츠(최소 버전)
   - 마이
   - 알림함 / 알림 설정
   - 푸시 초기화 및 라우팅
2. FastAPI backend
   - `/health`, `/api/v1/health`
   - 홈 / 종목 / 관심종목 / 테마 / 알림 / 디바이스 토큰 API
   - 관리자 로그인 및 관리자 CRUD API
3. admin_web
   - 관리자 로그인
   - 대시보드
   - 종목 / 가격 레벨 / 상태 / 이벤트 / 홈 노출 / 테마 / 콘텐츠 / 감사 로그 조회 및 저장
   - 수동 푸시 발송
4. signal batch
   - 신호 자동 생성 배치
   - 중복 생성 방지
   - 일부 종목 실패 시 전체 배치 지속 처리
5. push dispatcher
   - pending 알림 발송
   - no token / invalid token / failed / sent 상태 처리

아래 12개 항목을 모두 만족해야 1차 출시 가능으로 판단한다.

- [ ] 앱 실행 직후 치명적 크래시가 없다.
- [ ] 홈 화면이 정상 로드된다.
- [ ] 종목 검색이 가능하다.
- [ ] 종목 상세 진입이 가능하다.
- [ ] 종목 상세 차트가 최소 placeholder 포함 정상 렌더링된다.
- [ ] 관심종목 등록/해제/알림 on/off 가 동작한다.
- [ ] 알림함 목록 조회와 읽음 처리 동작이 확인된다.

### 4-2. 운영/데이터 플로우

- [ ] 관리자 로그인과 주요 CRUD 가 가능하다.
- [ ] 관리자에서 종목/레벨/홈 노출/테마/콘텐츠 수정 후 앱 반영이 확인된다.
- [ ] signal batch 수동 실행 시 signal_event 생성 또는 dry-run 결과 확인이 가능하다.
- [ ] push dispatcher 실행 후 pending 알림 상태 전이가 확인된다.
- [ ] 운영 환경 분리(dev/staging/prod 또는 local/prod)가 문서대로 명확히 유지된다.

> 위 항목 중 하나라도 실패하면 출시 가능 판정을 보류한다.

## 5. 이슈 등급 분류 기준

### 5-1. Blocker

다음 중 하나라도 해당하면 blocker 다.

- 앱 실행 불가 또는 반복 크래시
- 홈 / 종목 검색 / 종목 상세 / 관심종목 중 하나라도 완전히 동작하지 않음
- 관리자 로그인 불가 또는 관리자 저장이 전혀 반영되지 않음
- backend 전체 장애 또는 주요 API 5xx 지속
- signal batch 가 전혀 실행되지 않거나 모든 종목에서 실패
- push dispatcher 가 알림을 전혀 처리하지 못함
- 운영 환경(dev/prod) 혼선으로 잘못된 서버를 바라봄
- 보안상 즉시 수정이 필요한 민감정보 노출

### 5-2. Major

출시는 가능하지만 출시 직후 빠른 수정이 필요한 수준이다.

- 일부 핵심 화면에서 빈 상태 / 오류 상태 처리가 부정확함
- 특정 기능은 동작하지만 데이터 갱신이 늦거나 재시도 UX 가 부족함
- 관리자 저장 후 앱 반영 시간이 비정상적으로 길거나 간헐 실패함
- 푸시가 일부 시나리오에서만 누락됨
- 배치가 일부 종목 오류를 과도하게 발생시킴
- 중요한 문구/상태 라벨이 사용자 오해를 유발함

### 5-3. Minor

기능은 동작하며 출시 차단 사유는 아니지만 품질 보강이 필요한 수준이다.

- UI 문구 다듬기
- 카드 정렬/간격/스타일 보완
- 빈 상태/에러 상태 디자인 보강
- 관리자 JSON 출력 가독성 개선
- 알림 문구/상세 설명 고도화

## 12. Cloud Firestore 읽기 비용 검토

현재 저장소는 Cloud Firestore를 사용하지 않는다.

즉, 이번 QA 체크리스트 문서 작성으로 인해 Firestore read 비용이 증가하는 변경은 없다.

실제 데이터 읽기 비용 관점에서는 backend API + PostgreSQL + FCM fallback/log 구조를 기준으로 점검하면 된다.
