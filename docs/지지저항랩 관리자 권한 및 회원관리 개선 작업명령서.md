# 지지저항랩 관리자 권한 및 회원관리 개선 작업명령서
버전: v1.0  
대상: Codex / Flutter 개발자  
목적: 로그인 시 `role`이 `guest`로 덮어써지는 문제를 해결하고, 관리자 화면에서 회원의 `guest / member / admin` 권한을 관리할 수 있도록 기능을 확장한다.

---

## 0. 이번 작업의 최우선 목표

1. 로그인/회원가입/프로필 동기화 시 기존 `users/{uid}` 문서의 `role`, `allowedPaths`가 덮어써지지 않게 수정한다.
2. 관리자 허브에서 진입 가능한 **회원관리 화면**을 추가한다.
3. 회원관리 화면에서 `users` 컬렉션의 계정을 조회하고, `role` 및 `allowedPaths`를 수정할 수 있게 한다.
4. Firestore Rules와 실제 앱 동작이 충돌하지 않도록 권한 구조를 정리한다.

---

## 1. 문제 정의

현재 증상:
- Firestore 콘솔에서 특정 계정의 `role`을 `admin`으로 변경해도,
- 사용자가 다시 로그인하면 `role`이 `guest`로 돌아간다.

추정 원인:
- 로그인 또는 프로필 저장 시점에 `users/{uid}` 문서를 무조건 `set` 하면서,
  기존 `role`, `allowedPaths`를 기본값(`guest`, `[]`)으로 덮어쓰고 있다.

영향:
- 관리자 계정 유지 불가
- 회원관리 기능을 만들어도 로그인 시 다시 권한이 초기화됨
- 보안상 매우 위험하며 운영 안정성이 떨어짐

## 3. 권한 정책

### role 정의
- `guest`
  - 기본 가입 사용자
  - 관리자 기능 접근 불가
- `member`
  - 일반 회원
  - 사용자 기능 허용
  - 관리자 기능 접근 불가
- `admin`
  - 관리자 허브 접근 가능
  - 운영 관심종목 수정 가능
  - 회원 권한 변경 가능

### allowedPaths 정의
- 선택적 세부 권한 제어용
- 예시:
  - `/admin`
  - `/admin/users`
  - `/admin/watchlist`

### 이번 단계 정책
- 관리자 접근 여부는 우선 `role == 'admin'` 기준으로 유지
- `allowedPaths`는 이후 세부 권한 확장용으로 함께 관리 가능하게만 둔다

## Part A. role 덮어쓰기 버그 수정

### 목표
기존 `users/{uid}` 문서가 있을 경우, 로그인 시 `role`, `allowedPaths`를 절대 기본값으로 덮어쓰지 않도록 수정한다.

### 구현 원칙
1. 최초 로그인/회원가입 시 `users/{uid}` 문서가 없으면 생성한다.
2. 이미 문서가 있으면:
   - 기존 `role` 유지
   - 기존 `allowedPaths` 유지
   - 필요한 최소 필드만 보정
3. 프로필 동기화 시에도 `role`, `allowedPaths`는 기본값으로 재설정하지 않는다.
4. `displayName`, `email`, `photoUrl`, `lastLoginAt` 등은 갱신 가능하다.
5. 문서 생성과 문서 갱신 로직을 분리한다.

### 해야 할 일
- [ ] `auth_repository.dart`에서 로그인 후 사용자 문서 생성/갱신 흐름 점검
- [ ] `guest` 기본값이 매 로그인마다 들어가는 코드 제거
- [ ] 기존 문서 존재 여부 확인 후 분기 처리
- [ ] 최초 생성 시에만 `role: 'guest'`, `allowedPaths: []` 설정
- [ ] 기존 문서 갱신 시 `role`, `allowedPaths` 보존
- [ ] 가능하면 `lastLoginAt` 필드 추가 또는 갱신
- [ ] 닉네임/이메일 미존재 시에만 보정하도록 안전하게 수정

### 완료 기준
- [ ] Firestore 콘솔에서 `role: admin` 설정 후 재로그인해도 유지된다
- [ ] `allowedPaths` 값도 재로그인 후 유지된다
- [ ] 신규 가입자는 기본 `guest`로 정상 생성된다

## 8. 구현 시 주의사항

1. 로그인 시 `set(..., merge: true)`를 써도, payload에 `role: 'guest'`가 포함되면 결국 덮어써진다.  
   따라서 merge 여부보다 **payload 내용 자체**를 분리해야 한다.

2. 사용자 프로필 보정 로직과 권한 로직은 분리해야 한다.  
   예:
   - 프로필 보정: `displayName`, `photoUrl`, `lastLoginAt`
   - 권한 보정: 하지 않음

3. 회원관리 화면에서 저장할 때도, 불필요한 필드를 전체 덮어쓰기하지 말고 필요한 필드만 업데이트한다.

4. Firestore 문서 누락 필드가 있어도 앱이 죽지 않게 null-safe 파싱이 필요하다.

5. allowedPaths 입력은 쉼표/공백/줄바꿈 기반 문자열을 배열로 정리해서 저장한다.
