## 3. Flutter 앱 환경 분리

Flutter 앱은 이미 `AppConfig.fromEnvironment()` 구조를 사용하므로, 이번에는 `--dart-define` 기준으로 운영 환경을 명확히 분리한다.

### 3-1. 사용되는 값

- `APP_ENV`: `dev` / `staging` / `prod`
- `API_BASE_URL`: 앱이 호출할 API base URL
- `USER_IDENTIFIER`: 로컬/테스트용 사용자 식별자
- `ENABLE_VERBOSE_LOG`: 개발 로그 허용 여부
- Firebase 관련 define 들

### 3-2. 실행 예시

#### dev

```bash
flutter run \
  --dart-define=APP_ENV=dev \
  --dart-define=API_BASE_URL=http://127.0.0.1:8000/api/v1 \
  --dart-define=USER_IDENTIFIER=demo-user \
  --dart-define=ENABLE_VERBOSE_LOG=true
```

#### staging

```bash
flutter build apk \
  --dart-define=APP_ENV=staging \
  --dart-define=API_BASE_URL=https://staging-api.example.com/api/v1 \
  --dart-define=ENABLE_VERBOSE_LOG=true
```

#### prod

```bash
flutter build appbundle \
  --dart-define=APP_ENV=prod \
  --dart-define=API_BASE_URL=https://api.example.com/api/v1 \
  --dart-define=ENABLE_VERBOSE_LOG=false
```

### 3-3. 운영 원칙

- 운영 앱은 반드시 `APP_ENV=prod` 와 prod `API_BASE_URL` 을 함께 사용한다.
- release 빌드에서 dev URL 이 들어가지 않았는지 마지막으로 재확인한다.
- Firebase 프로젝트도 환경별로 분리하는 것을 권장한다.

## 4. backend 환경변수 방식

backend 는 `backend_api/app/core/config.py` 의 `Settings` 를 통해 환경변수를 읽는다.

### 4-1. 주요 환경변수

- `APP_ENV`
- `APP_DEBUG`
- `APP_VERSION`
- `LOG_LEVEL`
- `DATABASE_URL`
- `SECRET_KEY`
- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`
- `ADMIN_JWT_SECRET`
- `CORS_ORIGINS`
- `CORS_ORIGIN_REGEX`
- `SCHEDULER_ENABLED`
- `SIGNAL_BATCH_ENABLED`
- `PUSH_ENABLED`
- `FCM_ENABLED`
- `FCM_SERVICE_ACCOUNT_JSON`
- `FCM_SERVICE_ACCOUNT_FILE`
- `FCM_PROJECT_ID`
- `FCM_SERVER_KEY`

### 4-2. `.env.example` 준비

`backend_api/.env.example` 를 복사해서 각 환경별 `.env` 를 만든다.

```bash
cd backend_api
cp .env.example .env
```

운영에서는 저장소에 실제 `.env` 를 커밋하지 않는다.

### 4-3. CORS 설정

- `CORS_ORIGINS` 는 쉼표 구분 문자열로 입력한다.
- dev 는 localhost 계열을 허용한다.
- staging / prod 는 실제 배포 도메인만 넣는다.
- 운영에서는 전체 와일드카드 사용을 피한다.

예시:

```env
CORS_ORIGINS=https://admin.example.com,https://app.example.com
CORS_ORIGIN_REGEX=
```

## 5. health endpoint / 운영 확인 포인트

### 5-1. 엔드포인트

- `GET /health`
- `GET /api/v1/health`

### 5-2. 확인 값

- `status`
- `environment`
- `version`
- `database`
- `scheduler_enabled`
- `signal_batch_enabled`
- `push_enabled`

### 5-3. 기대 결과

- DB 연결이 되면 `status=ok`, `database=ok`
- DB 연결 실패 시 `status=degraded`, `database=error`

## 6. admin_web 환경 분리

admin_web 는 정적 파일이라 서버 사이드 env 주입이 없으므로 `admin_web/assets/env.js` 를 배포 환경마다 다르게 둔다.

### 6-1. 파일 구조

- 실제 사용: `admin_web/assets/env.js`
- 예시 템플릿: `admin_web/assets/env.example.js`

### 6-2. 예시

```js
window.__SRLAB_ADMIN_CONFIG__ = {
  appEnv: 'prod',
  apiBaseUrl: 'https://api.example.com/api/v1',
  adminUsername: 'admin',
};
```

### 6-3. 운영 원칙

- admin 비밀번호는 절대 `env.js` 에 넣지 않는다.
- admin 은 기본 API base URL 만 주입하고, 인증은 backend 환경변수 계정으로 처리한다.

## 7. migration / seed / bootstrap

### 7-1. migration

```bash
cd backend_api
alembic upgrade head
```

배포 순서상 backend 프로세스 공개 전에 먼저 실행하는 것을 권장한다.

### 7-2. 개발용 seed

```bash
cd backend_api
python scripts/seed_minimum_data.py
```

- dev / local 검수용으로 사용한다.
- prod 에서는 샘플 데이터 대량 주입 용도로 사용하지 않는다.

### 7-3. admin bootstrap

현재 MVP 는 별도 관리자 테이블 대신 환경변수 기반 관리자 계정을 사용한다.

즉, 최초 운영 계정 bootstrap 은 아래 값 준비로 완료된다.

```env
ADMIN_USERNAME=admin
ADMIN_PASSWORD=<strong-password>
ADMIN_JWT_SECRET=<strong-secret>
```

## 8. batch / worker 실행 방법

### 8-1. signal monitor batch

```bash
cd backend_api
python -m app.tasks.run_signal_monitor --dry-run
python -m app.tasks.run_signal_monitor
```

권장:
- dev/staging 에서 먼저 `--dry-run`
- prod 에서는 dry-run 검수 후 실제 실행
- 중복 실행 방지를 위해 한 환경에서 하나의 스케줄러만 운영

### 8-2. notification dispatcher

```bash
cd backend_api
python -m app.tasks.run_notification_dispatcher --limit 50 --max-retry-count 3
```

### 8-3. 운영 스케줄 예시

- signal monitor: 장중 1~5분 간격 또는 운영 정책에 맞춘 cron
- notification dispatcher: 1분 간격 cron 또는 worker loop

### 8-4. 주의사항

- prod 에서는 여러 서버에서 같은 cron 이 동시에 돌지 않게 한다.
- 배치 장애 시 먼저 환경변수 / DB / push credential 을 확인한다.

## 9. 권장 배포 순서

### 9-1. backend

1. 환경별 `--dart-define` 로 빌드
2. 홈 / 검색 / 종목상세 / 관심종목 기본 흐름 확인
3. prod 빌드가 dev 서버를 보지 않는지 재검토

## 11. rollback 최소 절차

1. `GET /health` 실패 여부 확인
2. 최근 변경(코드 / env / migration) 중 무엇이 먼저 달라졌는지 확인
3. 필요 시 batch/scheduler 일시 중지
4. 직전 안정 릴리스로 backend 를 되돌림
5. env 오타 / 누락 수정
6. admin 로그인 / 홈 API / 종목상세 API 재검증
7. 앱은 기존 안정 빌드를 유지하고 새 배포를 중단

### migration 관련 주의

- 파괴적 migration 이 예정되어 있으면 반드시 사전 백업을 수행한다.
- MVP 단계에서는 가능하면 additive migration(컬럼 추가 등) 위주로 운영한다.
